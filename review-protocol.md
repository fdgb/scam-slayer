# 审核协议（Review Protocol）· Scam Slayer

> 本文件是「营销号亲爹」所有数据采集、投稿审核、知识库入库的**统一执行标准**。
> 设计哲学：**系统不假装全知**。审核的目标不是"判断真假"，而是"评估可信度 + 呈现证据"，把最终判断权交还给人。

---

## 0. 一条核心原则

作者本人、自动化、甚至大模型，都**不可能 100% 判断任何内容的真假**。
因此本项目的所有结论都必须是**带证据的分级提醒**，而不是终审判决。

- 不确定时，标 🟡（不确定），**绝不**为了"显得专业"而强行标 🔴。
- 误报（把真消息判成假）比漏报更伤信任，必须优先避免。

---

## 1. 来源分级（信任上限）

每条样本/案例必须先定来源等级，等级决定它**最高能进哪一层**：

| 等级 | 来源举例 | 信任上限 |
|------|----------|----------|
| **🅢 S 级 · 官方权威源** | 微信辟谣/腾讯较真、国家反诈中心(96110)、各地网警/反诈中心、中国互联网联合辟谣平台、公安部/网信办、市场监管总局 | 可进「已确认模式库」 |
| **🅐 A 级 · 权威媒体** | 新华社、人民网、光明网、央视新闻、中新网、省消保委/市场监管通报 | 可进「已确认模式库」 |
| **🅑 B 级 · 一般来源** | 其他媒体转述、自媒体、论坛、网友投稿 | **只能**进「可疑特征库」，不得作为确定结论依据 |

---

## 2. 交叉验证闸门（入库前的硬门槛）

一条内容要被标为「已确认骗局模式」，必须**同时通过**两关：

1. **≥2 个独立来源相互印证**
   - 至少含 1 个 S/A 级原始出处（警方通报、公司官方声明、官媒报道），而非只有自媒体转述。
   - "独立"指来源之间无引用关系（不是同一篇稿子的互转）。
2. **反向核查无权威反驳**
   - 主动搜索"是否存在权威辟谣 / 官方澄清 / 当事方否认"的反面证据。
   - 若找到有权威来源的反驳 → **降级**或标「争议」，不得标已确认。

未通过闸门的，按证据强度落入「可疑特征库」或「待核实」。

---

## 3. 三层置信入库

| 层 | 落入条件 | 是否进公开库 | 用户侧用途 |
|----|----------|--------------|------------|
| **已确认模式库** | S/A 级 + 通过交叉验证闸门 + 无反驳 | ✅ 进 `data/` | 作为 🔴 疑似结论的硬依据 |
| **可疑特征库** | 有红 flag，但来源弱 / 仅单源 / 存争议 | ✅ 进 `data/` | 仅作 🟡 不确定的提示素材 |
| **待核实** | 信息不足 / 来源互相矛盾 / 反向核查有强力反驳 | ❌ 不进，留 `pending_samples.md` 暂存 | 不对外输出 |

---

## 4. 条目字段规范（每条必带）

入库条目统一带以下字段，便于下游检索与置信判断：

```
类型:     营销号 / 谣言 / 诈骗 / AI伪造 / 其他
平台:     抖音 / 微信 / 小红书 / 微博 / 电话短信 / 其他
置信度:   高 / 中 / 低
依据:     [来源机构名] [URL] [采集/核查日期]
红flag:   1~3 条具体、可核对的特征
规律总结: 一句话抽象出的"模式/特征"（不点名具体账号）
```

- 低置信条目：结论部分只写"疑似/存疑特征"，**不得**写"确认是骗局"。
- 所有 `依据` 必须可核查，禁止编造来源链接。

---

## 5. 红线（合规 + 防误伤）

- **只沉淀"模式/特征"，绝不点名具体真人/账号就是骗子。**
- 涉及具体受害人、机构时须脱敏或省略，不采个人身份信息。
- 涉时政类内容格外谨慎，只谈公开可核的骗局手法，不评价是非。
- 不自动 push 未审核内容；所有采集先入 `pending_samples.md` 暂存。

---

## 6. 与自动化的衔接

- **每日采集自动化**（每天运行）：按本协议的 §1 给每条打 S/A/B；按 §4 带字段；按 §2 对"拟标已确认"的条目做反向核查；简报写明 S/A/B 分级数量。
- **周一审核入库自动化**（每周一运行）：读取 `pending_samples.md`，逐条按 §2 交叉验证 → 按 §3 分层 → 写入 `data/` → 重建 `version.json` → commit + push；在 `pending_samples.md` 顶部标记已入库、避免重复。
- **Expert 输出**（用户来问时）：套用「诚实分级输出协议」五段式（见 SKILL.md），与本条 §0–§3 一一对应。
